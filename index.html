Building an AI-Assisted Exam Preparation System
with Claude Code

EASTD143B Digital Tools and Methods in East Asian Humanities: Coding Approach  ·  Spring 2026

I.Background & Motivation

  • General Exam: 3 fields, 220+ sources, cross-linguistic materials (Chinese / Tibetan / English)
  • What the exam actually demands — not recall, but real-time argument production:
    • Given a question, immediately assemble a defensible thesis from the right sources
    • Deploy evidence in a logical sequence under pressure
    • Anticipate and answer follow-up challenges from three different advisors, each with a distinct intellectual style
  • That means the system needs to encode three kinds of logic simultaneously:
    • My thinking logic — how I read sources, what I think matters, where I see tensions
    • Exam logic — what counts as a strong argument, how to structure it, what a defensible thesis looks like
    • Advisor logic — what each examiner prioritizes, how they push back, what they find convincing
  • Conventional notes record content — they don't encode any of these three logics
  • This constraint directly shapes every design decision that follows — including what goes in as raw material (→ Section IV)
The goal is not a query layer on top of my notes — if I wanted to know what a given scholar argues, my notes are already clear enough for that. What I needed was a synthesized knowledge structure: one that already embodies argument logic, cross-source connections, and exam deployment strategy, so that under pressure I can retrieve not just information, but a complete line of reasoning.

II.Why This Tool Stack

  • A multi-tool system — each tool does a different job:
Tool Role
Notion Primary database — connected to other Notion databases; table, filter, and relational features I still rely on
Claude Project Note-taking stage — set up with exam background, full bibliography, and note requirements as persistent context; used for chapter selection guidance, note feedback, and revision before notes are exported
Claude Code
  • Builds and modifies the wiki — reads/writes markdown files directly, no intermediary layer
  • Runs mock exam sessions — simulates advisor-style questions to stress-test arguments before the real exam
  • After each mock session, updates the wiki with answers I couldn't defend or new insights worth keeping
Obsidian Views the wiki Claude Code generates — graph view, internal links, local file browsing
Git Version control — every edit is traceable; connects to GitHub for backup and sharing
  • The key decision: markdown flat files as the synthesis layer — format that both AI and humans can read and write directly, without being locked into any single platform's structure
  • Choosing markdown over a database means the structure of knowledge is determined by argument logic, not platform schema — the wiki looks the way it does because of how I think, not because of what Notion's columns allow

III.Getting Here: From Codex to Claude

  • Started with Codex — taught in class so convenient to use, and free with the Harvard account
  • Methodology felt right: feed notes → AI extracts and organizes → knowledge base
  • The problem: no matter how much I adjusted, rewrote rules, refined prompts — output was surface-level and random
    • My notes have many layers: arguments, debates, methodological moves, cross-text connections
    • Codex consistently only picked up the most obvious, most generic content
    • Spent significant time on prompt engineering — but couldn't trust what came out
  • Conclusion: my methodology isn't wrong — maybe it's the AI
  • Switched to Claude, using the exact same approach
  • One run in — immediately knew this was something I could actually use
  • Claude costs money. But for what this system needs to do, that's a price worth paying — you get what you pay for.

IV.The Raw Material

  • A synthesized knowledge structure can only be built from material that already contains the thinking — feeding AI raw texts would produce content extraction, not argument logic
  • Input is my reading notes and topic notes: they already embed my interpretive decisions — what matters, where the tensions are, how sources speak to each other
  • AI is therefore working with pre-interpreted material, not extracting patterns from raw text — this is the core design choice that makes the rest of the system possible
  • Language choice is a matter of efficiency: Chinese is my most natural thinking language, so notes are written primarily in Chinese — key terms, names, and expressions are glossed in brackets with English and, where relevant, Tibetan or Sanskrit

Why not use the original source texts directly?

  • ~80% of my reading list is monographs — having AI extract directly from a full book is too risky and defeats the purpose
  • My reading workflow:
    • Give AI the intro + table of contents → AI recommends which chapters are most worth reading
    • I read those chapters myself and write two versions of notes:
      • Argumentative version — structured claims, scholarly debates, methodological moves
      • Capture version — fragmentary, close to the text, can copy-paste
    • Send both to AI for feedback → revise together → store in Notion
  • All note-taking happens inside a Claude Project — set up with my exam background, full bibliography, and note requirements as persistent context, so every feedback session already knows the exam stakes and how my notes should be structured
  • When ready to synthesize: export from Notion as .md → Claude Code generates wiki → review in Obsidian
  • Key principle: I do the intellectual work; AI assists at specific junctures — chapter selection, note feedback, wiki generation

V.Wiki Architecture

Demo
  • raw/reading-notes/ — source notes (one per text, named by citekey)
  • raw/topic-notes/ — thematic notes that synthesize across multiple sources
  • wiki/topics/ — what I'm studying: sects, figures, institutions, historical events
  • wiki/concepts/ — analytical tools: theoretical frameworks and mechanisms used to explain the topics
  • wiki/debates/ — scholarly disagreements with named positions and stakes
  • wiki/methodology/ — one page per scholar: how they work, what their method can and cannot do
  • wiki/questions/ — my own open questions, linked to relevant topics and debates
  • wiki/cross_field/ — systematic cross-field comparative arguments, built for synthesis questions
  • AGENTS.md — persistent ruleset telling Claude how to behave in this workspace
  • The separation maps directly onto the three logics from Section I: topics = what the exam asks about; concepts + methodology = the analytical vocabulary I bring to it; cross_field = where exam logic and advisor logic are encoded as ready-to-deploy synthesis
  • Keeping them separate means any combination is possible on demand — a question about sacred geography can draw from topics, concepts, methodology, and cross_field simultaneously without any single page needing to contain everything

VI.Iterative Development with AI

  • Phase 1: Ask AI to turn notes into wiki pages → organized, but not exam-ready
  • Phase 2: Added an exam argument template to every core page
    • Opening thesis (one sentence you can say immediately)
    • Three-step deployment: anchor → develop → complicate & land
    • Key terms that must appear  ·  Anticipated follow-up questions
  • Phase 3: Designed /exam mock exam mode — three advisor personalities, six command formats
  • Phase 4: Built cross-field comparison pages for synthesis problems
  • Phase 5: Automated reading-note processing pipeline
    • YAML auto-generation — when a new note arrives without front matter, Claude generates a complete YAML block (citekey, author, year, field, topics, status, bibliography) rather than requiring manual entry
    • Harvard Library Catalog integration — Claude calls a custom skill to query HOLLIS and fill in the Chicago-style bibliography field automatically; falls back to note content if not found
    • Citekey rules — modern scholars: LastName+Year (e.g., Davidson2005); classical texts: conventional abbreviation or translator+year (e.g., HZSJ, Roerich1949); if year unknown: LastName+n.d.
    • Topic enrichment — Claude doesn't just copy the original Notion tags; it reads the full note and adds related historical, religious, and methodological topics the tags missed
    • Automatic file renaming — after citekey is confirmed, the file is immediately renamed to citekey.md; eliminates Notion's truncated-title + hex-string exports

VII.Challenges & Solutions

Challenge Solution
AI synthesis distorted nuanced arguments Human verification at every stage; I review every page AI generates before it enters the wiki
AI structures my misreadings and makes them look authoritative When errors caught, correct the source note first, then update the wiki — fixing at the root, not just the surface
AI cannot always pinpoint the right material when the request is a thinking problem, not a precise query — "find sources related to this line of reasoning" leaves wide interpretive room; what gets pulled in may be adjacent but not exactly what the argument needs Regular review cycles: go back through wiki pages, check whether the logic holds, fill gaps, and revise — the wiki improves through use, not just through generation
Didn't know which direction to prepare Incorporated advisor profiles — different argument structures for different examiners
Cross-field synthesis is hard to systematize Dedicated cross-field pages with comparison matrices

VIII.Reflections: Advantages & Limitations

  • Advantages
    • Because input is pre-interpreted, AI output directly reflects argument logic — not generic summaries extracted from raw text
    • Human-AI loop runs at every stage: chapter selection → note feedback → wiki generation → mock exam → review and revision; AI is never operating alone
    • Advisor personalities are encoded — mock exam sessions are calibrated to specific intellectual styles, not generic "practice questions"
    • AGENTS.md as a persistent ruleset: consistent behavior across sessions, with built-in anti-hallucination protocols and source-tracing requirements
    • Clear quality criterion at every stage: does the output survive advisor-style pushback in a mock session?
  • Limitations
    • N=1 — output quality depends entirely on the quality of my notes; the system cannot compensate for a weak or misread source
    • AI cannot verify sources independently: a well-structured error is harder to catch than an obvious one — the anti-hallucination protocol in AGENTS.md reduces but does not eliminate this
    • When requests are reasoning problems rather than precise queries, AI selection has significant interpretive latitude — requires ongoing human review to catch misalignments
    • What is "digitized" here is a cognitive process, not a text corpus — there is no shareable dataset, no reproducible pipeline independent of my specific notes and thinking
    • The system may reinforce existing thinking rather than extend it — if the wiki only structures what I have already reasoned through, the output risks becoming an echo chamber with no genuinely new critical moves; how to build more intellectual friction into the /exam mode is an open problem still being worked on
Most DH projects have AI find patterns in external texts, and humans interpret the results. This project inverts that: I interpret first, AI structures my interpretations, then AI stress-tests them. This isn't corpus analysis — it's cognitive scaffolding. Does that count as DH?

IX.Takeaways

  • Structuring knowledge for AI is itself a form of knowledge production — the process surfaces things I thought I understood but couldn't actually articulate
  • AI as thinking partner vs. data processor — two distinct modes of human-AI collaboration; the second is underrepresented in DH discourse
  • An epistemological question the project raised: when you structure your knowledge so AI can help you navigate it, does that change what "knowing" means?
